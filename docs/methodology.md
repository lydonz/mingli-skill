# 分析方法与输出约定

## 分层原则

本项目将输出拆为三个层次：

1. **历法层**：公历/农历、节气、干支、四柱、大运和流月边界。
2. **结构层**：五行、十神、旺衰、合冲、喜忌和紫微宫位等结构化信号。
3. **解释层**：事业、财运、关系或健康倾向的传统命理推演。

前两层可以通过后端和回归样例验证；第三层必须保留不确定性，不能当作现实事件预测。

## 推荐工作流

完整分析应按以下顺序进行：

1. 收集公历/农历、时间精度、地点、性别和问题范围。
2. 记录采用标准时间还是真太阳时，并检查是否跨越时辰或换日边界。
3. 输出四柱、藏干、十神、五行、大运和当前日期区间。
4. 以 `strength_assessment` 读取旺衰与冲突标记；不要将兼容字段 `legacy_strength` 用作结论依据。
5. 先分析月令、季节和五行结构，再看十神、格局和合冲关系。
6. 动态问题按“原局—大运—流年—流月”顺序展开；具体事件必须给出日期区间。
7. 紫微结果作为独立的辅助信号，和八字信号分别列示，不因单一系统产生确定结论。
8. 列出支持结论的结构化证据、相反信号、模型冲突和仍需现实验证的事项。
9. 将建议落到可执行的现实行动，如准备材料、沟通、预算、休息和寻求专业支持。

## 流月规则

流月不按公历月初切换。本项目使用小寒、立春、惊蛰、清明、立夏、芒种、小暑、立秋、白露、寒露、立冬和大雪十二个节气边界。

`analysis_period` 使用 `granularity: "month"` 或 `"day"` 时，`liunian.流月分段` 会输出每一段的开始与结束时刻、流年、流月、相对日主的十神和当期大运。

## 紫微审计与子时约定

紫微结果由 `iztro` 后端产生。`ziwei_raw` 保留十二宫、主星/辅星详情、亮度、生年四化、十二长生、十年大限以及 `紫微审计` 元数据；兼容摘要 `ziwei` 不得用来替代完整宫位数据。

`birth_context.zi_hour_convention` 可取 `benchmark`、`early` 或 `late`。`benchmark` 保持旧接口在 23:00 使用晚子、00:00 使用早子的映射；另外两项显式声明 23:00 至 00:59 的子时口径。该值必须进入 `birth_time`、紫微调用和 `chart_id`，以防止两份不同紫微盘共享同一审计标识。

HTML 报告中的十二宫视图只消费结构化宫位字段，并对所有星曜与宫位文本转义。视图用于检查排盘输出，不作为传统解释的证据。

## 解释文本契约

`interpretation_document` 是排盘完成后可选附加的文本层。它必须使用 `interpretation-v1`，携带当前 `chart_id`，并使每个段落引用 `interpretation_brief.allowed_evidence_ids` 中至少一项证据，同时提供 `uncertainty`。

契约只校验证据可追溯性、命盘一致性和字段边界，不能证明解释文本或未来事件预测正确。所有报告中的解释字段按纯文本 HTML 转义；校验失败必须以 `component_status.interpretation` 可见状态返回，且不得影响历法、紫微或规则引擎结果。

## 规则建议边界

`rules_suggestion` 是面向历史选择题兼容的可选组件，不是自由问答结论。没有选项时它返回 `rules_suggestion_no_options`，不执行随机或猜测性选择。

健康、伤害、法律类别，以及问题或选项中出现死亡、医疗、投资、借贷、赌博、破产、怀孕或离婚等高风险主题时，组件返回 `degraded` 抑制状态。此时仍可返回排盘、流年区间、数据质量和传统文化结构化资料，但不得产出 A/B 预测。

## 本地知识包

本项目将知识资料与历法计算、规则引擎完全分离。`knowledge_packs/` 中的每个包必须有 `knowledge-pack-v1` 清单，其中声明上游仓库、固定提交、许可证、文件角色和 SHA-256。检索器只搜索通过校验的本地 Markdown 文件，不进行网络请求。

`HybridMingliToolkit.analyze_question()` 仅在调用方传入 `knowledge_query` 或 `knowledge_packs` 时才检索。返回的 `knowledge_references` 每项都包括知识包标识及版本、上游来源、文件、章节、行号、摘录和用途标记；`component_status.knowledge` 与 `knowledge_context.pack_statuses` 公开包损坏、缺失或无匹配的状态。

知识包可以解释传统术语、分析顺序和流派分歧，但不得：

- 修改四柱、节气、紫微或大运的计算结果。
- 覆盖 `strength_assessment`、规则证据或模型冲突。
- 被描述为未来事件、健康、财富、职业或关系结果的验证证据。

## 外部排盘回归

`DestinyLinker/MingLi-Bench` 的固定提交
`b7433280fd86d7a7c27debbc47d0303c218f0bfd` 被限定为排盘回归来源。其
`fortune_api_results.json` 中的 32 份 `iztro` 命盘可比较四柱分量、农历日期、
命身宫、五行局、生肖和十二宫主辅星结构。

该数据集的 160 道选择题描述的是历史事件。导入器不得保存题干、选项或答案，
评估器不得计算事件答案命中率，也不得将其用于训练、宣传或推断事业、财富、
健康、关系等现实结果。

紫微对照双方都使用 `iztro`，它的作用是发现本项目适配层、字段映射和依赖版本
的回归，不应被当作独立算法或天文历法的验证。

## 报告规则

HTML 报告来自同一份综合 JSON 输出，因此会保留：

- 命盘标识和出生时间校正。
- 四柱、五行、旺衰、大运、流年和流月。
- 事业、财富、关系和健康模块的结构化摘要。
- 组件状态、数据质量提示和免责声明。
- 当调用方明确请求并获得本地知识引用时，附带来源、章节、行号和文化参考边界。

报告不会加载远程脚本、字体、图片或追踪服务。公开报告前应先移除可识别个人的信息。
